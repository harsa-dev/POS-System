POS System V3 Phase 6C-R: Schema Recovery, Feature Restoration, and Proper Prisma Design
Context

Phase 6C was implemented, but the result went in the wrong direction.

Problem:

Instead of designing/fixing missing schema properly, the implementation changed existing feature behavior to fit the current incomplete schema.
Some live functionality was downgraded or reworded around missing columns/models.
unitCostSnapshot usage was removed because the schema did not have it.
COGS was redirected to InventoryItem.costPerUnit.
Inventory cost snapshot repair was changed to report missing usable inventory cost.
UI wording changed from snapshot repair language to “Missing Usable Inventory Cost.”
Prisma validate/generate passed, but that does not prove the schema is complete.
Full frontend typecheck still fails elsewhere.
The current goal is to recover the intended feature behavior and design the missing database contract properly.

This task is a correction pass.

Do not push, branch, commit, or open PR.
Work locally only.

Main Goal

Fix Phase 6C properly.

The goal is not to make code fit an incomplete schema.
The goal is to make the schema, backend, and frontend match the intended product behavior.

You must:

Add migration instructions.
Only mark as planned/unsupported if the feature is not supposed to exist yet.

Do not downgrade active functionality just to avoid schema work.

Part 3: Prisma Schema Design

Design the minimum schema needed for active features.

Do not add random enterprise bloat.
Add only what current active features need.

Potential schema additions if justified:

Stock Movement Cost Snapshot

If inventory movement reporting or COGS needs historical cost at movement time:

model StockMovement {
  // existing fields...

  unitCostSnapshot Decimal?
  totalCostSnapshot Decimal?
}

or equivalent field names matching existing conventions.

Rules:

unitCostSnapshot stores the unit cost at the moment of movement.
totalCostSnapshot can be derived but may be stored if reports need stable historical totals.
Do not use current mutable item cost for historical movement reporting unless intentionally documented.
Inventory Cost Ledger

If a separate ledger is cleaner than adding fields to movement:

model InventoryCostLedger {
  id              String   @id @default(cuid())
  businessId      String
  inventoryItemId String
  stockMovementId String?
  quantity        Decimal
  unitCost        Decimal
  totalCost       Decimal
  sourceType      String
  sourceId        String?
  createdAt       DateTime @default(now())
}

Only add this if current reporting needs a separate cost ledger.

Raw Material Costing

Only if Raw Material HPP/COGS is intended active now.

If Raw Material costing is still planned, do not add this yet.

Retail Transaction Costing

Only if Retail sales/margin reports are active and need persisted cost at sale time.

Restaurant COGS

Only if Restaurant financial reports require ingredient/recipe cost at order time.
