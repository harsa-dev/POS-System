Where things actually stand today Shared Dashboard The single-resolver architecture (DashboardShell as the one place that picks a mode adapter) was completed in Phase 6B — that's good structural work. But the full frontend TypeScript check (tsconfig.json) still fails — registry/sidebar metadata types, stale role unions, shared card prop drift, dashboard panel prop mismatches. Phase 6B itself flags this as the immediate next task. 7+ shared surfaces (Financial Reports, Invoice Generator, Customers & Partners, Inventory, Team Management, Audit Log, Approvals) have not yet adopted the BM-9 mode-context contract — they're still showing base/generic data regardless of which mode is active. Restaurant — fully production-ready. No urgency here. Minor leftovers: shift cashier workspace, non-cash payment handoff. Retail — active foundation, partial API wiring. Cashier, receiving, returns: real API. Stock opname, shelf management, promotions: interactive mocks with no backend endpoints yet. Raw Material — core persisted (suppliers, batches, intakes, weighings, kandang). Cashflow, HPP, Financial Reports, Invoice: correctly hidden as "unsupported" until procurement schema exists. Why shared dashboard first 1. The TypeScript failure is a structural foundation problem, not a surface bug. Every mode depends on the shared registry/sidebar/dashboard panel contracts. Retail and Raw Material cores are built on top of these types. Adding real APIs to Retail while the shared type layer is drifting just defers a harder merge later. 2. Fixing shared once benefits all three modes simultaneously. Rolling the BM-9 mode-context contract out to the remaining 7 surfaces is formulaic work — the pattern is proven (Cashflow already did it). One batch of work, three modes improved. 3. A broken shared surface undermines trust in mode switching. When a user switches from Restaurant to Retail and opens Financial Reports or Team Management, they see wrong data (base generic content, not Retail-scoped). This breaks the whole "modes feel like separate workspaces" promise — regardless of how good the Retail cashier is. 4. Mode cores have hard schema blockers that take more design time. Retail's missing APIs are self-contained but need backend schema + route + frontend wiring per surface. Raw Material's missing models (procurement, material master, cost ledger) are explicitly deferred in Phase 4B because they require schema design before any UI work. Shared dashboard fixes don't need new schema. Proposed sequence Phase A — Shared dashboard foundation (do this first) Fix the full tsconfig.json typecheck: resolve registry/sidebar contract drift, stale role unions, shared card/panel prop mismatches Roll out the BM-9 mode-context contract to the 7 remaining shared surfaces so each one correctly scopes to the active mode Goal: switching modes and opening any shared dashboard shows the right data, no TypeScript errors in the full check Phase B — Retail core (second) Wire real backend endpoints for stock opname, shelf management, promotions Clean up the as unknown as Prisma delegate casts in Retail persistence Goal: Retail has zero interactive mocks in its primary workflow surfaces Phase C — Raw Material core (third) Design and implement the minimal procurement + material master schema (RawMaterialPurchase, material master with min stock + avg cost) Unlock cashflow, HPP/COGS, supplier invoice surfaces for Raw Material Goal: Raw Material feels like a complete operational mode, not just reads The one case for mode core first If the priority is "make something feel shippable to a specific type of business user as fast as possible," Retail is the closest to completable (Restaurant is already done, Raw Material needs schema work). You could argue: get Retail to 100% first, then clean the shared foundation. That's a valid business-first argument if you have a Retail customer waiting. But architecturally, shared dashboard first is the lower-risk, higher-leverage path.
